## 当前 mapping 建议小修

### 1. `信息维护/资料信息修改` 不应映射到 `card_rebinding`

当前表：

```json
{
  "intent": "信息维护/资料信息修改",
  "skill_ids": ["card_rebinding"]
}
```

实际唯一 golden 样本是 record 281，客户问的是紧急联系人查询/修改：

> “我这边想查询我的紧急联系人……紧急联系人可能换了联系方式……机主已经换人了”

这不是换绑还款银行卡。建议二选一：

- 新增 `contact_info_maintenance` / `profile_info_maintenance` 类 skill 后映射过去。
- 如果暂时不新增 skill，则该 intent 应标为当前能力缺口，不应继续用 `card_rebinding` 兜底。

这是当前 mapping 里最明确的一处错配。

### 2. `申请咨询/预约借款` 当前样本更像 `loan_termination`

当前表：

```json
{
  "intent": "申请咨询/预约借款",
  "skill_ids": ["loan_consultation"]
}
```

实际唯一 golden 样本 record 285：

> “早上提交了一笔借款，到现在没批下来，我不需要了，给我取消掉，我不贷了。”

这和 `loan_termination` 的触发词“取消借款 / 不借了 / 取消申请”更一致。建议至少改为：

```json
"skill_ids": ["loan_termination"]
```

如果担心 `预约借款` 未来会回到活动咨询语义，可以临时写成：

```json
"skill_ids": ["loan_termination", "loan_consultation", "value_added_service_inquiry"]
```

但从当前 golden 评测口径看，`loan_termination` 最干净。

### 3. 催收/提醒类应补 `close_pre_reminder`

当前 `close_pre_reminder` 有独立 SOP 和 skill，但没有被 mapping 使用。golden 中已经存在“未逾期/还款日提醒/频繁电话/要求关掉”的样本，例如：

- record 5：未逾期但一直打电话，客户要求关掉。
- record 288：未逾期两天打十几个提醒电话，客户投诉。
- record 234：还款日当天/未逾期提醒、联系人收到短信，客户要求不要乱发。

建议：

```json
"催收相关/要求停催": ["stop_collection", "close_pre_reminder"]
"催收相关/投诉催收": ["collection_complaint", "close_pre_reminder"]
```

这样不会把逾期后停催和预提醒混为同一个生产 skill，但评测上能承认预提醒 SOP 是合理命中。

### 4. `还款相关/存对公还款` 建议补 `repayment_result_query`

当前表接受：

```json
["repayment_method_inquiry", "repayment_status_issue", "overpayment_refund", "post_loan_verification"]
```

但 27 个对公样本里有一批是“已经对公转账/公户还款后，想确认是否入账、是否还在欠款、是否继续催收”。这类 query 可能被路由到 `repayment_result_query`，而不是 `repayment_status_issue`。

建议改为：

```json
[
  "repayment_method_inquiry",
  "repayment_status_issue",
  "repayment_result_query",
  "overpayment_refund",
  "post_loan_verification"
]
```

## 标注维度全量扩展建议

这些二级小结没有出现在当前 golden，但在 `标注维度.xlsx` 中存在。它们不应全部塞入当前正式评测表，而应单独维护覆盖状态。

| 标注二级小结 | 当前 SOP/skill 状态 | 建议 |
|---|---|---|
| `业务办理/逾期还款证明` | 有 `other_certificate` | 若未来进入评测，映射到 `other_certificate` |
| `申请咨询/贷款解约` | 有 `loan_termination` | 若未来进入评测，映射到 `loan_termination` |
| `营销活动/会员咨询` | 有会员/优享卡/增值服务咨询 skill | 可映射到 `member_consultation`、`premium_card_inquiry`、`value_added_service_inquiry` |
| `营销活动/营销电话` | 有 `stop_marketing` | 可映射到 `stop_marketing`，必要时补营销承诺未兑现分支 |
| `营销活动/退费进度` | 没有专门增值服务退费进度 skill | 建议新增 `value_added_refund_status`，临时不建议混到费用退费进度 |
| `催收相关/特殊场景` | 部分可由 `overdue_negotiation` / `stop_collection` / `close_pre_reminder` 承接 | 标签太粗，需按三级小结拆 |
| `产品与信息/下载或登录` | 当前无明确 SOP/skill | 若要评测需新增登录/下载类 skill |
| `产品与信息/注册申请` | 当前无明确 SOP/skill | 若要评测需新增注册申请类 skill |
| `产品与信息/公司信息` | 当前无明确 SOP/skill | 若要评测需新增公司信息咨询 skill |
| `产品与信息/信息安全` | 当前无明确 SOP/skill | 若要评测需新增信息安全/非我司客户处理 skill |
| `产品与信息/产品建议1` | 当前无客服业务 SOP/skill | 建议标为非当前业务评测范围 |
| `批量问题/*` | 无通用业务 skill | 建议标为批量事件/运营标签，不进 skill recall 评测 |
| `自定义/*` | 非业务意图标签 | 建议排除出 skill recall 评测 |

## 推荐执行顺序

1. 当前正式 mapping 已扩展到主要二级分类，并补入 `close_pre_reminder`、`repayment_result_query`、`loan_termination` 等承接关系。
2. 对 `信息维护/资料信息修改` 仍需做产品决策：新增资料/联系人维护 skill，或继续将该标签标成当前能力缺口。
3. Demo 侧已保留 `skill_to_intents`，避免只用单一 `skill -> intent` 反向表而丢失一对多/多对一关系。
4. 后续如果 `scripts/references/merged_intent_skill_mapping.json` 继续调整，应重新校验 `config/eval_intent_mapping.json` 和 `/api/demo/eval/intent-options` 的展开结果。
